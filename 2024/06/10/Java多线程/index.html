Java多线程
chengkx Lv2
  2024-06-10 21:08:47   2024-10-22 17:36:50

一、线程

1、一个基本的CPU执行单元 & 程序执行流的最小单元

注意：

  1. 比进程更小的可独立运行的基本单位，可理解为：轻量级进程
  2. 组成：线程ID + 程序计数器 + 寄存器集合 + 堆栈
  3. 注：线程自己不拥有系统资源，与其他线程共享进程所拥有的全部资源。

2、作用

减少程序在执行时付出的时空开销，提高操作系统的并发性能

3、状态

三种：就绪、阻塞、运行

二、线程分类

主要分为守护线程和非守护线程

守护线程：

守护用户线程（非守护线程）的线程，如垃圾回收线程

非守护线程：主线程和子线程

主线程（UI线程）

  • 定义：Android系统在程序启动时会自动启动一条主线程
  • 作用：处理四大组件与用户进行交互的事情（如UI、界面交互相关）
  • 注：因为用户随时会与界面发生交互，因此主线程任何时候都必须保持很高的响应速度，所以主线程不允许进行耗时操作，否则会出现ANR

子线程（手动创建的线程）

网络请求、IO操作等等、耗时的线程

守护线程和非守护线程的区别：

区别：虚拟机是否已退出：

  • 当所有用户线程结束时，因为没有守护的必要，所以守护线程也会终止，虚拟机也同样退出；
  • 反过来，只要任何用户线程还在运行，守护线程就不会终止，虚拟机就不会退出

三、线程优先级

主要分10级，一般默认情况下是5级

四、多线程

  1. 其实，计算机任何特定时刻只能执行一个任务；
  2. 多线程只是一种错觉：只是因为JVM快速调度资源来轮换线程，使得线程不断轮流执行，所以看起来好像在同时执行多个任务而已

1、主线程和子线程（ 工作线程）

  1. 不要阻塞UI线程（即主线程）：单线程会导致主线程阻塞，然后出现ANR错误：主线程被阻塞超过5s则会出现错误
  2. 不要在UI线程之外更新UI组件

所以一般是一个UI线程和X个工作线程‘

将耗时任务放在工作线程中：不要阻塞UI线程（即主线程），即当我们有耗时的任务，如果在UI线程中执行，那就会阻塞UI线程了，必须要抛到工作线程中去执行；

将UI组件的更新内容放在UI线程中：不要在UI线程之外访问UI组件，即更新UI组件时，一定得在UI线程里执行，故需要在工作线程中执行的任务结果返回到UI线程中去更新组件

2、实现方式

大致上来说分为两种：

继承Thread类

实现Runnable接口

基本线程池（暂时空出）

五、线程调度

时间片轮转的方式调度，所以线程不可能绝对同步

  • 在采用时间片的系统中，每个线程都有机会获得CPU的资源以便进行自身的线程操作；当线程使用CPU资源的时间到后，即时线程没有完成自己的全部操作，JVM也会中断当前线程的执行，把CPU资源的使用权切换给下一个队列中等待的线程。

被中断的线程等待CPU的下一次轮回，从中断处继续执行

调度的优先级根据线程优先级执行

进程 状态说明

就绪-运行

处于就绪状态的进程调度后，获得处理及资源，转为运行状态

运行-就绪

执行完毕，让出处理机资源

运行-阻塞

进程请求某一资源的使用和分配时

进程等待某一事件的发生

进程和线程

进程：资源拥有的基本单位、使程序能够并发执行，提高系统的资源利用率和吞吐量、进程的地址空间之间相互独立、拥有资源、通信复杂、系统开销大

线程：独立调度和分派的基本单位、降低程序并发执行的时空开销，提高操作系统的并发性能、拥有很少的资源，某个进程下的所有线程共享该进程的资源、系统开销很小、通信简单

六、线程同步

1、Synchronized关键字

作用：保证只有一个线程执行该方法/代码块

使用规则：

锁对象的设置：a.修饰代码块时，需要一个reference对象，作为锁的对象

​ b.修饰实例方法时，默认的锁对象=当前对象

​ c.修饰类方法（静态）时，默认的锁对象=当前类的class对象

根据锁对象的不同，一把锁同时只能被一个线程持有：若目标锁已被当前线程持有，则其他线程只能阻塞等待当前线程释放目标锁

若当前线程已持有目标所，其他线程仍可以调用目标类中没有被synchronized修饰的方法

当对象获取多个锁时，必须以相反的顺序释放&在与所有锁被获取时相同的词法范围内释放所有锁：若线程进入由线程已拥有的监控器保护的synchronized块，就允许线程继续进行，只有线程退出它进入的监控器保护的第1个synchronized块时，才释放锁

注意：java类中，实例对象会有多个，但是只有1个class对象：即类的不同实例，共享该类的class对象，实际上，class对象也属于java对象

静态方法&实例方法上的锁默认不一样，若同步则需制定两边锁，静态方法加锁，能和所有其他静态方法加锁的进行互斥

锁的等级和类型

synchronized会修饰不同的内容，例如代码块、类的实例方法&静态方法

类型：对象锁、方法锁、类锁

对象锁：含有synchronized方法/代码块的类的实例对象

使用前必须先获得对象的锁，线程进入synchronized方法时获取该对象的锁，若此对象的对象锁已经被其他调用者占用，则需要等待此锁被释放再进入。

Java的所有对象都含有1个互斥锁，这个锁由jvm自动获取和释放

synchronized方法正常返回or抛异常终止，JVM会自动释放对象锁体现了用synchronized来加锁的第一个好处：方法抛异常时锁仍然可以由JVM来释放

方法锁：使用synchronized修饰的方法

synchronized方法控制对类成员变量的访问

每个类实例对应1把锁

每个synchronized方法都必须获得调用该方法的类实例的锁才能执行，否则所属线程阻塞

方法一旦执行，就独占该锁，直到从该方法返回时才能将锁释放，此后被阻塞的线程才能获得该锁，重新进入可执行状态

所有声明为synchronized的成员函数中至多只有一个处于可执行状态，从而有效避免了类成员变量的访问冲突。

类锁：使用synchronized修饰静态方法/代码块

控制静态方法之间的同步

一个静态方法被声明为synchronized。此类中的所有实例化对象在调用此方法共用一把锁

一个类不论被实例化多少次，其静态方法和变量在内存中都只存在一份

缺陷：若修饰一个大方法，将大大影响效率

  • 若使用Synchronized关键字修饰 线程类的run()，由于run()在线程的整个生命期内一直在运行，因此将导致它对本类任何Synchronized方法的调用都永远不会成功
  • 解决方案
    使用 Synchronized关键字声明代码块
