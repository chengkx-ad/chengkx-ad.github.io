📚定义传输格式：

服务端需要什么信息，才能唯一确定客户端需要调用的接口呢?

🐇RpcRequest 定义调用方法的信息

服务器调用这个方法以后，需要给客户端返回哪些信息呢？

🐇RpcResponse 请求号 响应状态码

通过网络传输告诉服务端我要调用这个接口，服务端接收到后，找到这个接口的实现类，并且执行，将执行的结果返回给客户端，作为客户端调用接口方法的返回值。

🌱注册多个服务

📚在前面的工作中，我们设计了一个简单的RPC框架，在服务器测试中，注册完helloService服务后，服务器就自行启动了。意味着一个服务器只能注册一个服务。接下来，我们要将服务的注册和启动分离，使得服务器可以提供多个服务。

🐇服务注册表

需要一个容器，作用：保存本地服务信息，获得一个新服务名字的时候能够返回该服务的信息。

创建一个ServiceRegistry接口：getService获取服务信息、register注册服务信息

🐇实现该接口：DefaultServiceRegistry

在该类中，主要作用是实现上面的接口（提供注册服务）。ConcurrentHashMap<>存储服务信息，服务名与提供服务的对象的对应关系

下面这个set集合，用来存储有哪些服务被注册过

在注册服务时，默认采用这个对象实现的接口的完整类名作为服务名。之后想再获得服务的对象就去map里查

📚一些其他处理：

🐇RpcServer 在创建RpcServer时要传入注册好服务的ServiceRegistry，服务的注册不再需要RpcServer处理，主负责启动
🐇RequestHandlerThread：处理线程，从ServiceRegistry获取到提供服务的对象

在每一个请求处理线程中也需要传入serviceRegistry了

在RequestHandlerThread中处理线程，从ServiceRegistry获取到提供服务的对象

RequestHandler：接到RpcRequest和提供服务的对象，反射调用

📚实现NettyServer和NettyClient

NettyClient

note：在静态代码块中就配置好了客户端，只需要等待发送数据时启动，channel将RpcRequest对象写出，并且等待服务端返回的结果。此处发送是非阻塞的，所以发送后会立刻返回，从而无法得到结果。所以我们以阻塞的方式返回结果

Netty设计模式—–责任链模式

责任链上有多个处理器，对数据进行加工，处理完成后将数据传给下一个处理器。

🐇自定义协议

首先是 4 字节魔数，表识一个协议包。接着是 Package Type，标明这是一个调用请求还是调用响应，Serializer Type 标明了实际数据使用的序列化器，这个服务端和客户端应当使用统一标准；Data Length 就是实际数据的长度，设置这个字段主要防止粘包，最后就是经过序列化后的实际数据，可能是 RpcRequest 也可能是 RpcResponse 经过序列化后的字节，取决于 Package Type。

🐇解码器
🐇数据处理器
🐇序列化接口

序列化方法主要用了四种：Hessian算法、Json、Kyro、protobuf。

🐇NettyServerHandler 和 NettyClientHandler

NettyServerHandler 和 NettyClientHandler 都分别位于服务器端和客户端责任链的尾部，直接和 RpcServer 对象或 RpcClient 对象打交道，而无需关心字节序列的情况。

NettyServerhandler 用于接收 RpcRequest，并且执行调用，将调用结果返回封装成 RpcResponse 发送出去。
至此，我们已经实现了使用netty传输的RPC框架，自己定义了传输协议，构建了一个通用的序列化框架。

其中，kyro序列化器是我们默认的序列化方式。
